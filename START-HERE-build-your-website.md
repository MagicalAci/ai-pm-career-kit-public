# Start Here: 让 Cursor 像产品教练一样带你做网站

如果你现在脑子里只有一个模糊想法，或者你只知道“我想做个网站”，那就从这份文档开始。

这份文档的目标不是讲方法论，而是让你真的往前推进，最后把网站搭出来。

## 第一步：先把仓库拉下来

```bash
cd ~/Downloads && git clone https://github.com/MagicalAci/ai-pm-career-kit-public.git && cursor ~/Downloads/ai-pm-career-kit-public
```

如果你只想安装 skill 到自己的 Cursor：

```bash
mkdir -p ~/.cursor/skills && git clone https://github.com/MagicalAci/ai-pm-career-kit-public.git /tmp/ai-pm-career-kit-public && rm -rf ~/.cursor/skills/ai-product-0to1-playbook && cp -R /tmp/ai-pm-career-kit-public/.cursor/skills/ai-product-0to1-playbook ~/.cursor/skills/ && rm -rf /tmp/ai-pm-career-kit-public
```

## 第二步：把这段话直接发给 Cursor

复制下面整段，不要拆开：

```text
你现在进入“产品教练模式”。

我的目标不是只讨论想法，而是从一个模糊想法开始，最后真的搭出一个网站。

请优先使用 `ai-product-0to1-playbook` skill，并且像产品教练一样带我走完全流程。

无论我接下来给你的想法多模糊，你都要按下面的方式工作：

1. 先判断我们当前处于哪个阶段
2. 在每次回复开头先告诉我：
   - 当前阶段
   - 当前总进度
   - 已完成什么
   - 现在最该做什么
3. 一次只推进一个最关键步骤，不要一口气把所有框架都倒给我
4. 每一阶段都要告诉我：
   - 这一阶段的目标
   - 我需要补充什么信息
   - 你会产出什么
   - 这一阶段完成的标准是什么
5. 如果我的信息不够，你先问我 3-5 个最关键的问题
6. 你的最终目标不是写文档本身，而是把网站做出来

我当前的信息如下：
- 产品名：[你的产品名]
- 一句话介绍：[一句话介绍]
- 我现在的想法：[把你脑子里的想法直接写出来]
- 我猜的目标用户：[目标用户]
- 我觉得他们的问题：[核心问题]
- 当前阶段：[想法 / 调研 / 原型 / 已准备开始做网站]
- 我的资源限制：[时间 / 预算 / 会不会写代码 / 有没有设计能力]

现在请你先判断我们在哪个阶段，并只推进第一步。
```

## 第三步：你接下来应该看到什么

一个合格的“产品教练式” Cursor，不应该直接丢给你一大堆空话。它应该每次都先告诉你：

- 当前阶段：例如 `1/15 用户分群`
- 当前进度：例如 `已完成 0，正在开始 1`
- 现在你要做什么
- 它接下来会交付什么

如果它没有这样做，你就直接补一句：

```text
请按产品教练模式回复：先说当前阶段、当前进度、我现在要做什么、你会产出什么，再开始展开。
```

## 你会经历的完整流程

| 阶段 | 你现在要做什么 | Cursor 应该给你什么 | 完成标准 |
|---|---|---|---|
| 1. 用户分群 | 说清你觉得谁最可能需要这个网站 | 用户段、JTBD、首批切入用户建议 | 已明确第一批用户 |
| 2. 产品战略 | 确认这个网站到底解决什么问题 | 价值主张、边界、北极星、取舍 | 已明确做什么和不做什么 |
| 3. 验证实验 | 决定是否值得做 | XYZ 假设、验证实验、成功阈值 | 已知道先验证什么 |
| 4. 竞品分析 | 看别人怎么做、哪里还能切 | 竞品格局、差异点、定位建议 | 已有差异化说法 |
| 5. PRD | 把方向写清楚 | 一份结构化 PRD | 已有统一需求文档 |
| 6. 用户故事 | 把需求拆细 | 用户故事、验收标准 | 已能进入执行颗粒度 |
| 7. 原型规划 | 决定页面和流程 | 页面列表、screen plan、状态设计 | 已能画原型或直接开做 |
| 8. AI 能力设计 | 如果网站里有 AI 功能，定义怎么做 | Prompt、输入输出、兜底策略 | AI 能力不是一句空 prompt |
| 9. 测试场景 | 提前想清楚怎么验收 | 测试步骤、边界场景、smoke test | 已有 QA 思路 |
| 10. 指标设计 | 确定网站上线后看什么 | NSM、转化指标、健康指标 | 知道怎样判断网站有用 |
| 11. GTM | 想清楚怎么让人知道 | 渠道策略、信息表达、上线节奏 | 不是做完就算了 |
| 12. 网站范围 | 决定 v1 网站最小包含什么 | sitemap、页面清单、必须模块 | 已有明确页面范围 |
| 13. 网站文案 | 把价值写到页面上 | 标题、副标题、模块文案、CTA | 页面内容可直接写进去 |
| 14. 网站构建计划 | 准备真正开始搭 | 技术栈、组件、页面实现顺序 | 能进入开发 |
| 15. 上线 QA | 上线前做最后检查 | QA、埋点、SEO、部署清单 | 网站可上线 |

## 每个阶段你只需要做什么

### 阶段 1 到 4：先把方向讲清楚

你只需要把你的想法尽量用人话讲出来，不需要一开始就很专业。

比如你可以这样开口：

- 我想做一个帮 `[某类人]` 解决 `[某个问题]` 的网站
- 我也不确定这是不是个好方向，你先帮我收一下用户和场景
- 我脑子里现在只有几个零散想法，你先别写 PRD，先帮我判断问题对不对

### 阶段 5 到 8：把方向变成可执行方案

这时候你要开始接受一件事：不再只是聊想法，而是要把页面、功能、AI 能力、输入输出都说具体。

你可以这样对 Cursor 说：

- 现在我们进入执行阶段，请帮我先把网站的一期范围定下来
- 先不要讲长远愿景，先把网站首页、功能页、FAQ、CTA 这些最小范围列出来
- 如果网站里有 AI 功能，请把 prompt、输入输出和失败兜底一起设计

### 阶段 9 到 15：把网站真的做出来

从这一步开始，你的目标就变成“上线”，不是“继续讨论”。

你可以这样说：

- 现在不要再发散，直接进入网站落地模式
- 请给我网站文案、页面结构、技术栈建议和实现顺序
- 请把结果写到我可以直接交给工程或直接用 AI coding 开做的程度
- 每次只告诉我当前最应该完成的一步

## 如果你脑子里只有一个模糊想法

那也没关系，直接这样说：

```text
我现在只有一个很模糊的想法，目标是做一个网站，但我不确定用户是谁、价值是什么、先做哪些页面。请你像产品教练一样，不要一次讲完全部框架，而是先判断我现在在哪个阶段，再一步一步带我往前走。
```

## 你怎样知道自己没有跑偏

每推进一阶段，至少检查这 4 件事：

1. 我是不是知道当前阶段叫什么
2. 我是不是知道这一阶段的产出是什么
3. 我是不是知道现在最该做哪一件事
4. 我是不是拿到了可以往下一阶段推进的明确结果

如果这 4 个问题里有两个答不上来，说明你现在还在“聊想法”，没有真正推进。

## 最后提醒

- 不要一开始就让 Cursor 直接写网站代码，先把方向收清楚
- 不要一直停在战略层，阶段 12 之后必须进入网站范围、文案和构建计划
- 不要害怕你的想法很乱，产品教练模式本来就是为了从混乱走到清晰
- 你真正的目标不是写一堆文档，而是让网站上线
